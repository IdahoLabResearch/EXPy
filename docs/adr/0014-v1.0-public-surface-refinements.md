# v1.0 Public-Surface Refinements: Namespace Rename, Feature Gating, CLI Parity, Typed Exception Attributes, Envelope Split

**Supersedes ADR-0006 on the encode/decode error contract for unsupported roots and on what is part of the exception contract. Supersedes ADR-0012 on the semver MAJOR-bump list (additions: exception-attribute contract, Namespace enum member rename or removal).**

Pre-v1.0 grilling surfaced five refinements to the public surface as it currently stands in code. Each is in-scope before tagging because shipping the current shape and changing it post-tag would force a MAJOR bump on the next change in the same area:

1. **`ProtocolEnum` is renamed to `Namespace`** (matching CONTEXT.md's resolved glossary, where Namespace is the dispatch unit and Protocol is the V2G standard a Namespace belongs to). The eight member names (`SAP`, `DIN`, `ISO2`, `ISO20_COMMON`, `ISO20_AC`, `ISO20_DC`, `ISO20_WPT`, `ISO20_ACDP`) and their string values (used to derive the `.so` filename) are unchanged. The `EXIProcessor.__init__` keyword argument renames `protocol: ProtocolEnum` → `namespace: Namespace`. No deprecated `ProtocolEnum` alias is shipped; AcCCS (the only named consumer per ADR-0002 and ADR-0004) updates in its own repo.
2. **Six-method `EXIProcessor` API is feature-gated.** Today `encode_fragment`, `decode_fragment`, `encode_xmldsig`, `decode_xmldsig` exist as methods on every Processor instance and raise `NotImplementedError` at call time on Namespaces whose libcbv2g schema doesn't define the corresponding root (today: SAP and DIN, which have no `*_exiFragment` and no `xmldsigFragment`). Under v1.0 these methods are **absent** on instances whose schema doesn't define the root: `hasattr(p, "encode_fragment")` returns `False` and direct attribute access raises `AttributeError`. Feature detection becomes honest and `hasattr`-based instead of try/except-against-`NotImplementedError`-based.
3. **CLI binaries gain four modes with the same feature gating.** Each of the eight `*Processor` binaries (ADR-0012) currently supports `--encode` and `--decode` (Document root). Under v1.0 each gains `--encode-fragment`, `--decode-fragment`, `--encode-xmldsig`, `--decode-xmldsig`, dispatched through the existing `extern "C"` entry points the Python wrapper uses. The new flags are **absent from `--help`** on Namespaces whose schema doesn't define the corresponding root (SAP, DIN — for both Fragment and XmldsigFragment); passing an unsupported flag exits with a clear Namespace-specific error rather than a generic parse failure.
4. **`EncodeError` and `DecodeError` carry typed attributes.** Today consumers have to substring-match `str(e)` to distinguish a libcbv2g buffer overflow (`rc=-200`) from an EXPy marshaler-input rejection (`rc=-1000`). Under v1.0 both exceptions carry typed attributes `rc: int`, `namespace: str`, `root: Literal["exiDocument", "exiFragment", "xmldsigFragment"]`. The `str(e)` format from ADR-0006 is preserved for human-readable logs but **marked informational** — consumers discriminating on failure mode use the attributes, not message substring matching.
5. **Top-level message envelope shape varies per Namespace** and that variation is documented as a known property of the surface. DIN and ISO-2 wrap top-level messages in `V2G_Message(Header=..., Body=...)` (the schemas define a `V2G_Message` complex type). SAP and the five ISO-20 Namespaces have no `V2G_Message` and instead use raw `{"<MessageName>": ...}` or `{"Body": {"<MessageName>": ...}}` envelopes at the top level. V2Gjson constructors do not paper over this — they reflect each Namespace's schema verbatim. The README documents the split so consumers see it explicitly.

Additionally, **V2Gjson identifier naming continues to mirror libcbv2g verbatim**, including non-PEP-8 casing (`costKindType`, `RelativeTimeIntervalType`, `EVSE_NotReady`). Identifier casing-only changes are not part of the MAJOR-bump rename rule of ADR-0012; the README documents the convention and recommends per-file `ruff` `N801,N802,N806` suppressions for strict-linting consumers.

## Considered Options

- **Keep `ProtocolEnum`; rename only CONTEXT.md.** Rejected: CONTEXT.md was the place we resolved the ambiguity, and shipping v1.0 with the code still using the loose term locks the loose term in for a MAJOR bump cycle. The cost of the rename is mechanical; the cost of carrying the inconsistency is permanent until 2.0.
- **Ship a deprecated `ProtocolEnum = Namespace` alias.** Rejected: no known external consumer imports the class by name (AcCCS updates in its own repo per ADR-0002/0004 cross-references), and an alias signals that the loose term is still acceptable — exactly the signal CONTEXT.md was written to reject.
- **Keep `NotImplementedError`-at-call-time for unsupported roots.** Rejected: it forces consumers to write try/except code paths for what is fundamentally a capability question. `hasattr` is the right idiom and the underlying ctypes-symbol check already runs at `EXIProcessor.__init__` time, so feature-gating doesn't introduce new failure modes.
- **Keep the CLI two-mode and document Fragment/XmldsigFragment as Python-only.** Rejected: the CLI was added to ADR-0012 precisely so consumers without Python integration can drive every codec path, and PnC workflows (signature computation, contract certificate install) live in Fragment / XmldsigFragment. Shipping the CLI without those modes means the shell-script surface is strictly weaker than the Python surface.
- **Pure substring-match exception contract.** Rejected for v1.0 — the ADR-0006 message format was never designed as a parseable API. Promoting `rc`/`namespace`/`root` to typed attributes makes the discrimination first-class without breaking the human-readable message; consumers reading the message are not broken, consumers parsing the message are nudged toward the attributes.
- **Auto-wrap SAP and ISO-20 messages in a synthetic `V2G_Message`.** Rejected: the EVerest JSON shape (ADR-0003/0006) is "what libcbv2g consumes and produces." Synthesizing an envelope that doesn't exist in the schema would diverge from libcbv2g and force a translation layer at every Namespace boundary. Documenting the split is the honest contract.
- **PEP-8 the V2Gjson identifiers.** Rejected for v1.0: a global rename across thousands of generated symbols would break every existing call site, force a translation table between libcbv2g headers and Python names, and only addresses lint warnings — not behavior. Per-file lint suppressions are the documented escape hatch.

## Consequences

- `ProtocolEnum` is renamed to `Namespace` in `EXIProcessor.py`; the `EXIProcessor.__init__` parameter renames `protocol` → `namespace`; the internal `self._namespace` attribute name happens to be correct already and is unchanged. ADRs 0002, 0003, 0004, 0006, 0012 are immutable historical record and retain their `ProtocolEnum` references — this ADR names the current contract going forward.
- `EXIProcessor.__init__` conditionally binds `encode_fragment` / `decode_fragment` / `encode_xmldsig` / `decode_xmldsig` to the instance only when the corresponding ctypes symbol exists. The class-level method stubs that today raise `NotImplementedError` are removed (or the methods are implemented as instance-bound closures gated by the symbol check — both are acceptable, choose the simpler). `tests/integration/test_namespace_parity.py` and `tests/integration/test_unsupported_methods.py` flip their expectation from `pytest.raises(NotImplementedError)` to `pytest.raises(AttributeError)` and `hasattr(...) is False`.
- The shared CLI driver (`src/common.cpp` or wherever cxxopts is wired) gains per-Namespace conditional flag registration driven by the codegen pipeline's existing knowledge of which Namespaces have which roots. `tests/integration/test_cli_binaries.py` (or a new `test_cli_fragment_modes.py`) gains smoke tests for Fragment and XmldsigFragment round-trip via each capable binary.
- `EncodeError` and `DecodeError` are extended with three constructor-set attributes: `rc: int`, `namespace: str`, `root: Literal["exiDocument", "exiFragment", "xmldsigFragment"]`. The constructor signature stays backward-compatible — the existing single-positional-argument `EncodeError("msg")` form continues to work. `_format_error` / `_decode_with` / `_encode_with` in `EXIProcessor.py` populate the three attributes alongside the existing message. `tests/integration/test_encode_errors.py` and `test_decode_errors.py` assert against the attributes rather than substring-matching the message.
- Per ADR-0006's contract on `NotImplementedError` for unsupported roots — that clause is superseded by point 2 above. The `str(e)` message format described in ADR-0006 is retained for human-readable logs but is no longer part of the contract consumers parse against.
- Per ADR-0012's semver MAJOR-bump list — that list is extended with: any change to the `EncodeError` / `DecodeError` typed attribute contract (attribute removed, type changed, attribute semantics changed); any `Namespace` enum member rename or removal. Additive changes (a new `Namespace` member for a new schema, a new attribute on an exception, a new optional CLI flag) remain MINOR per ADR-0012.
- The README documents `hasattr`-based feature detection as the supported idiom, the envelope-shape split (DIN/ISO-2 use `V2G_Message`, SAP/ISO-20 hand-build raw dicts), the V2Gjson naming convention, and the typed exception attributes. The CHANGELOG.md v1.0.0 entry enumerates this surface explicitly.
