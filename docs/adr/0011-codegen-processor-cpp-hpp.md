# Generate Processor cpp/hpp from libcbv2g headers (#23)

ADR-0001 scoped EXPy's code generator to the Marshalers file (`getJson_*` / `getDoc_*`). The per-Namespace `<Namespace>Processor.cpp` — the `extern "C"` boundary that wraps libcbv2g encode/decode calls and exposes the EVerest JSON shape (see ADR-0003) — was kept hand-written, on the reasoning captured in #3's discussion: building the Processor template before all eight Namespaces (`SAP`, `DIN`, `ISO2`, `ISO20_COMMON`, `ISO20_AC`, `ISO20_DC`, `ISO20_WPT`, `ISO20_ACDP`) existed risked baking in assumptions that the later ISO20 variants would force back out. With #4–#7 closed and all eight Processors hand-written and visible side-by-side, the template axes are now resolved against real divergences instead of guesses, and the Processor cpp/hpp pair joins the marshalers under `src/generated/`.

The eight hand-written Processors diverged along four axes:

1. **libcbv2g C prefix.** The encode/decode entry points and root types are named `<c_prefix>_<root>` where `<c_prefix>` is one of `appHand` (SAP), `din`, `iso2`, `iso20` (Common), `iso20_ac`, `iso20_dc`, `iso20_wpt`, `iso20_acdp`. Each cpp hard-coded its prefix in struct declarations and function calls.
2. **Marshaler prefix.** ADR-0001's marshaler emitter strips one underscore-separated namespace prefix when naming the C++ marshaler function (`din_CostType` → `getJson_CostType`). For ISO-20 sub-Namespaces the prefix only partially strips, so the marshalers end up named `getJson_acdp_exiDocument`, `getJson_ac_exiDocument`, etc.; for SAP / DIN / ISO-2 / ISO20_COMMON they are bare `getJson_exiDocument`. Each cpp hard-coded the per-Namespace marshaler name at every call site.
3. **Root-type set.** The libcbv2g header for each Namespace declares some subset of `<c_prefix>_exiDocument`, `<c_prefix>_exiFragment`, `<c_prefix>_xmldsigFragment`. Four Namespaces (`ISO2`, `ISO20_COMMON`, `ISO20_AC`, `ISO20_DC`) have all three; the other four (`SAP`, `DIN`, `ISO20_WPT`, `ISO20_ACDP`) have only `exiDocument`. The hand-written cpp/hpp pair per Namespace declared and defined exactly the matching subset of `decode/encode_FromPythonBytes` / `decode/encode_fragment_*` / `decode/encode_xmldsig_*` symbols.
4. **`V2G_Message` wrapper unwrap.** DIN and ISO-2's exiDocument marshalers emit `{"V2G_Message": {…}}` at the root because their XSDs declare `V2G_Message` as the document root. The EVerest JSON shape elides that wrapper: decode strips `["V2G_Message"]` before returning to Python; encode wraps the caller's JSON in `{"V2G_Message": …}` before invoking `getDoc_exiDocument`. SAP, all five ISO-20 Namespaces, and the Fragment / XmldsigFragment paths for DIN / ISO-2 do not have the wrapper. Each cpp hard-coded the unwrap as a per-file decision.

Axes 1, 2, and 4 are policy (closed-set per Namespace, not derivable from anything libcbv2g ships). Axis 3 is data (derivable from which root typedefs the parsed Datatypes header contains). The new emitter splits accordingly: a hard-coded policy table for the first three axes, a header-parse step for the fourth.

A fifth divergence — the cxxopts-driven CLI `main()` in SAP / DIN / ISO2 — was structurally identical across the three exec targets save for help-text strings. It is not Namespace-specific in any deeper sense; it consumes only the four `extern "C"` symbols every Processor exposes (`decodeFromPythonBytes`, `encodeFromPythonDict`, `free_decoded_data`, `free_encoded_data`). It is now a single source file linked alongside each Processor object, not part of the codegen template.

## Emitter

`tools/codegen/processor_emitter.py` exposes:

- `NamespacePolicy` (frozen dataclass): `namespace`, `c_prefix`, `marshaler_prefix`, `datatypes_header`, `encoder_header`, `decoder_header`, `marshalers_include`, `processor_hpp_filename`, `unwrap_v2g_message`.
- `NAMESPACES: dict[str, NamespacePolicy]`: the closed-set table for the eight Namespaces.
- `detect_root_types(spec_names, c_prefix) -> set[str]`: scans parsed TypeSpec names for `<c_prefix>_exiDocument`, `<c_prefix>_exiFragment`, `<c_prefix>_xmldsigFragment` and returns the subset present. Exact name matching avoids false positives between `iso20` and `iso20_acdp` etc.
- `emit_processor_cpp(policy, root_types) -> str` and `emit_processor_hpp(policy, root_types) -> str`: the two file bodies.
- `main(argv)`: CLI `--namespace <NS> --header <path> --out-cpp <path> --out-hpp <path>`. Both outputs are written from one invocation so Make's grouped-target syntax (`&:`) keeps the dependency single-fire.

The cpp template emits one `_decode_block` + `_encode_block` pair per root present, plus an unconditional `free_encoded_data` / `free_decoded_data` block. The block bodies interpolate `c_prefix` into libcbv2g symbols (`decode_<c_prefix>_<root>`), `marshaler_prefix` into Marshaler symbols (`getJson_<mp><root>`), and gate `["V2G_Message"]` (decode) / `wrapped["V2G_Message"] = inJson;` (encode) on `policy.unwrap_v2g_message and root == "exiDocument"`. Fragment and Xmldsig blocks never carry the unwrap regardless of policy — those roots have no wrapper in any Namespace's schema.

The hpp template emits a parallel set of declarations gated on the same root set, plus the two ABI structs (`decoded_data`, `encoded_data`) and the two free-function declarations. The structs are declared once per Processor hpp; consumers must not include two Processor hpps in the same translation unit (would trip ODR), which is why the shared CLI uses a local forward declaration rather than including a per-Namespace hpp.

## CLI

`tools/expy_cli.cpp` is a single Namespace-agnostic translation unit containing `main()`. It declares the four Processor `extern "C"` symbols inline (no Processor hpp included) so the same source links against any of the three exec targets. The Makefile compiles it once to `build/expy_cli.o` and links that object alongside `<Namespace>Processor.generated.o` and `common.o` for each of `build/SupportedAppProtocolProcessor`, `build/DINProcessor`, `build/ISO2Processor`. The shared-library targets (`build/lib-<Namespace>Processor.so`) do not link the CLI object — those are consumed by `EXIProcessor.py` via ctypes and have no `main()`.

The help text now reads "Encode and decode V2G EXI Documents for one Namespace" rather than per-Namespace strings — the binary's own filename identifies the Namespace, so the help text doesn't need to.

## Considered Options

- **Hand-write each of the eight Processor cpp/hpp pairs.** Status quo before this issue. Rejected: the eight cpp files were ~1980 LoC of near-identical boilerplate; the three CLI mains were ~750 LoC of identical control flow with three different help strings. Maintenance cost compounds at every libcbv2g re-pin (rename a root type → eight files to update) and at every #23-class refactor.
- **Per-Namespace YAML/TOML policy manifest under `tools/codegen/`.** Rejected: the eight-Namespace set is closed (V2G itself defines no more Namespaces in the relevant standards) and small enough that an external data file adds a new format to learn for no payoff. A Python dict in the emitter is the right shape for a closed enumeration.
- **Pass the policy axes as CLI flags from the Makefile.** Rejected for the same reason — the Makefile would become the source of truth for policy that doesn't otherwise live there. Keeping the emitter self-describing (one file lists what the eight Namespaces are and how they differ) keeps the documentation in the emitter, where it can be tested and grepped.
- **Template the CLI `main()` into the generated Processor.cpp for SAP / DIN / ISO2, conditional on a per-Namespace `emit_cli: bool`.** Rejected: the CLI is structurally identical across the three exec targets and not Namespace-specific in any deeper sense than "we kept building SAP / DIN / ISO2 as exec targets, not just as .so's, because we had hand-written CLIs in those files." Templating would have added a fifth axis (`emit_cli`) to the generator for boilerplate that's better expressed as a single shared source file linked at the exec target.
- **Drop the CLI exec targets entirely.** Rejected. No test depends on the binaries — they are dev convenience for ad-hoc hex encode/decode — but dropping a developer-facing surface without a deprecation pass is a public API change in the spirit of ADR-0003 even if not technically a Python API break. Preserving them with one shared source file is cheap.
- **Generate one `Processor.cpp.in` template and substitute via `sed` / `m4` at build time.** Rejected. The four-axis interpolation (c-prefix × marshaler-prefix × root-set × V2G_Message wrap) crosses both string substitution and conditional emission, and ADR-0001 already established Python as the codegen language. A second templating mechanism would split the codegen surface.

## Consequences

- `src/<Namespace>Processor.cpp` and the five hand-written `src/<Namespace>Processor.hpp` files are deleted; the corresponding `src/generated/<Namespace>Processor.generated.{cpp,hpp}` pairs are produced at build time. `src/` itself now contains only `common.cpp` plus the `generated/` subtree; the hand-written/generated split is reflected by directory.
- A `make clean` followed by `make all` from a fresh checkout rebuilds the full tree, including all eight Processor cpp/hpp pairs, all eight marshalers files, and the shared CLI object.
- Adding a ninth Namespace (if a future V2G standard introduces one) is now: (a) add an entry to `NAMESPACES` in `tools/codegen/processor_emitter.py`; (b) add five Make variables (`<NS>_GENERATED`, `<NS>_HEADER`, `<NS>_PROC_CPP`, `<NS>_PROC_HPP`, `<NS>_SRCS`) and a pair of `&:` Make rules; (c) add the per-Namespace `LIBS_<NS>` link set and the shared-lib / optional exec targets. No emitter code changes unless the new Namespace exercises a sixth divergence axis.
- A libcbv2g re-pin that adds or removes a root typedef in a Namespace's header propagates automatically through `detect_root_types`: the regenerated Processor cpp/hpp pair gains or loses the matching extern "C" surface without any policy or Make change. A re-pin that changes a C-prefix or renames a root would break all eight pipelines deterministically (the generated cpp would fail to compile against the renamed libcbv2g symbol), surfacing the divergence at build time rather than silently regressing.
- ADR-0001 is extended in spirit: code generation from `*_Datatypes.h` now covers the full Processor surface, not just the marshalers. ADR-0001's own consequences ("non-Marshaler parts […] are kept hand-written or templated separately") are realised as "templated separately" by this ADR — the `extern "C"` block and the V2G_Message wrap policy are now part of the emitter's output, not hand-maintained.
- ADR-0002 (Namespace-granular Processor split) is unchanged. Each Namespace still produces one `.so` and (for SAP / DIN / ISO2) one exec, each linking only its own libcbv2g archive(s). The codegen path does not consolidate Namespaces; it consolidates the per-Namespace *source* files.
- ADR-0003 (public API surface) is unchanged. The six-method `EXIProcessor` API, the EVerest JSON shape, and the `V2Gjson/*`-is-internal boundary are unaffected — generation happens entirely at the C++ layer, below the ctypes binding the Python API sits on.
- The DIN-side and ISO-2-side V2G_Message wrap behaviour, previously a per-file decision repeated in two of eight cpp files, is now a single boolean axis on `NamespacePolicy`. A future Namespace that also needs the wrap (none of V2G's current standards declare one) sets `unwrap_v2g_message=True`; nothing else changes.
- The 977-test integration suite continues to pass against the regenerated tree on a clean build. No `xfail` markers were introduced; no fixtures were modified. The eight per-Namespace integration test files (`test_<ns>_generated_*roundtrip.py`, `test_<ns>_tracer.py`, `test_sap_roundtrip.py`) exercised every Namespace's Processor against the same fixtures before and after the migration.
