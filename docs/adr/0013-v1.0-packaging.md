# v1.0 Packaging: Installable `expy` Python Package

For v1.0 EXPy ships as a `pip install`-able Python package named `expy`. The source-tree-vendored layout consumers see today — `EXIProcessor.py` and `V2Gjson/` at the repo root, `_version.py` discoverable only when the repo root is on `sys.path`, `.so` files loaded by `ctypes` via `os.path.abspath(__file__ + "/../build/lib-…so")` from a `./build/` directory that exists only after `make` — is reorganized into a single importable package:

```
expy/
    __init__.py
    _version.py
    processor.py          # was EXIProcessor.py
    v2gjson/
        __init__.py       # was V2Gjson/__init__.py
        common.py
        sap.py
        din.py
        iso2.py
        iso20_common.py
        iso20_ac.py
        iso20_dc.py
        iso20_wpt.py
        iso20_acdp.py
        # .so files shipped here as package data
```

A `pyproject.toml` with a PEP 517 build backend capable of driving the existing C make/CMake step (scikit-build-core is the chosen path) compiles the eight per-Namespace `.so` files and includes them inside the installed package directory as package data. The ctypes loader in `expy/processor.py` resolves the `.so` path relative to the installed package (via `importlib.resources` or `__file__`-relative lookup), not via a `./build/` path on disk. Top-level imports become `from expy import EXIProcessor, Namespace` and `from expy.v2gjson import din, iso2, …`; the `sys.path` shims in `tests/integration/conftest.py` (currently appending the repo root so `from _version import __version__` resolves) are removed in favor of standard package imports.

The current source-tree layout was never declared part of EXPy's public contract in any ADR — ADR-0003 and ADR-0012 enumerate the public surface (`EXIProcessor`, `ProtocolEnum`, encode/decode methods, V2Gjson constructors, CLI binaries) without claiming on-disk layout stability. Restructuring to the `expy/` package is therefore not a MAJOR-bump-breaking change under the semver rules of ADR-0012; it is the layout v1.0 ships in.

## Considered Options

- **Declare the current source-tree layout as the v1.0 contract** (vendor the repo, set `sys.path` to the repo root, run `make`, import `EXIProcessor`). Rejected: keeps EXPy un-`pip-install`-able, forces every consumer (AcCCS and any future tool) to either vendor or write a custom installer shim, and makes version pinning across consumers awkward. The `from _version import __version__` style only works with the repo root on `sys.path` — a packaging accident, not a designed surface.
- **sdist-only — ship source, build at install time, no wheels.** Rejected for v1.0: pure-sdist works on Linux with a toolchain present but fails opaquely on consumer machines without `make`/`cc`; wheel builds via scikit-build-core give us per-platform binary distribution when we want it without precluding source installs (`pip install .` from a checkout still works).
- **Pure-Python wrapper around a system-installed `libcbv2g.so`.** Rejected: libcbv2g isn't packaged for any Linux distro EXPy targets, our build applies several local libcbv2g patches (ADR-0007, ADR-0008, ADR-0009, ADR-0010) that aren't upstream, and the per-Namespace `.so` split (ADR-0002) is a build-time slicing of libcbv2g — there is no single system `.so` to wrap.
- **Drop the C compilation step into a separate build artifact and ship Python-only.** Rejected: the `.so` files are how EXPy works — the Python code is a ctypes wrapper. Splitting Python and native into separate packages just relocates the version-coupling problem and complicates installation.

## Consequences

- A `pyproject.toml` at the repo root drives the build via scikit-build-core (or an equivalent PEP 517 backend that wraps make/CMake) and bundles the eight `.so` files as package data under `expy/v2gjson/` (or directly under `expy/`).
- `EXIProcessor.py` moves to `expy/processor.py`; `V2Gjson/` moves to `expy/v2gjson/`; `_version.py` moves to `expy/_version.py`. The `from _version import __version__` cross-module pattern is replaced with `from expy._version import __version__` (or re-exported from `expy/__init__.py`).
- The ctypes loader in `expy/processor.py` resolves each per-Namespace `.so` via `importlib.resources` or `__file__`-relative lookup inside the installed package; the existing `./build/`-relative path is gone.
- `example.py`, README snippets, and all nine integration test modules update to `from expy import EXIProcessor, Namespace` and `from expy.v2gjson import …`. The `sys.path.append` shim in `tests/integration/conftest.py` is deleted.
- The Makefile is either retained as a convenience wrapper that invokes the pyproject build, or removed in favor of `pip install .` / `pip install -e .`. Whichever option ships, the README documents it as the supported build entry point.
- CLI binary outputs (the eight `*Processor` executables from ADR-0012) continue to land somewhere consumers can run them — either inside the wheel as console scripts, or in the project root after build, with the README documenting the new path.
- Per the rules above, the layout change itself is not a MAJOR break. Once v1.0 ships, `expy.` import paths and the `.so` data layout become part of the contract: subsequent rearrangement requires MAJOR.
